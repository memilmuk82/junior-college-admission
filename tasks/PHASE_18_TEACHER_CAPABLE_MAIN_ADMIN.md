# Phase 18 작업 카드 — 교사 기능을 겸하는 주 관리자와 운영 계정

## 목표

- 활성 주 관리자 계정이 계정 관리 권한과 함께 교사 학급·학생 성적·상담자료 업무를 수행한다.
- 주 관리자 소유 학급에 학생 계정을 연결하고 연결 범위에서 성적·상담자료를 공유한다.
- 주 관리자가 개인 BYOK 키를 계정 actor 범위에 암호화 저장하고 마스킹·교체·삭제한다.
- 사용자 확인용 비데모 주 관리자 계정을 운영 DB에 생성하고 기존 사용자 제공 OpenAI 키를 해당 계정에만 주입한다.

## 선행조건과 근거

- Phase 17 공개 검증과 네 역할 데모 배포가 완료되어 있어야 한다.
- `2027_전문대_입시상담앱_실행개발문서_v2.md`의 계정 분리·BYOK 암호화 원칙을 유지한다.
- `CODEX_MASTER_PROMPT_2027_전문대_입시상담앱_v3.md`의 단계 게이트와 운영 백업 절차를 따른다.

- 애플리케이션: `app/`
- 합성 익명 검증: `tests/`
- 검증·운영 자동화: `scripts/`, `Makefile`, `deploy/`
- 문서: `README.md`, `DEVELOPMENT_LOG.md`, `PROJECT_STATUS.md`, 이 작업 카드

## 금지 사항

- API 키 또는 새 계정 비밀번호를 소스·DB 평문 열·로그·테스트 결과에 기록
- 공개 데모 주 관리자에게 쓰기 권한 부여
- 보조 관리자에게 교사·주 관리자 기능 부여
- 기존 사용자 미추적 파일이나 원본 문서 변경
- 실제 학생 자료를 테스트 또는 저장소에 추가

## Given / When / Then 수용 기준

1. 활성 비데모 `ADMIN`이 로그인하면 주 관리자 메뉴와 교사 학급·학생 상담·BYOK 메뉴를 함께 본다.
2. 해당 관리자가 학급과 비식별 학생을 생성하면 학생 계정이 연결 코드로 연결할 수 있다.
3. 해당 관리자가 학생 성적과 상담자료를 저장하면 관리 주체가 그 관리자 계정으로 기록된다.
4. `TEACHER ↔ ADMIN` 역할 변경은 교사 업무 연속성을 보존하고, 교사 기능이 없는 역할 또는 비활성 전환은 기존 연결을 해제한다.
5. 공개 데모 주 관리자의 모든 교사 쓰기 요청은 계속 차단된다.
6. 운영 계정 BYOK는 암호문으로만 저장되고 화면에는 마스킹된 메타데이터만 표시된다.
7. 비데모 주 관리자 계정의 로그인·학급 화면·관리자 화면·BYOK 화면·로그아웃을 HTTPS Playwright로 확인한다.

## 먼저 작성할 실패 테스트

- 주 관리자의 교사 대시보드 메뉴와 학급 생성·학생 연결 서비스 테스트
- 주 관리자의 상담자료 저장·메모와 공개 계산 저장 테스트
- 교사/관리자 간 역할 변경 시 연결 보존 테스트
- 공개 데모 주 관리자 교사 쓰기 차단 회귀 테스트

## 남은 위험

- 주 관리자는 본래 전역 자료 관리 권한을 가지므로 교사 화면에서도 조회 범위가 넓다. 학급 생성·상담 저장의 소유자는 현재 로그인한 관리자 ID로 명시한다.
- 운영 임시 비밀번호는 배포 결과에서 한 번만 전달하고 저장소에는 남기지 않는다.
