# Phase 14 작업 카드 겸 Codex 실행 프롬프트: 엑셀 업무 흐름 중심 전면 재구축

이 파일 전체를 저장소 루트에서 Codex CLI 비대화형 실행의 프롬프트로 사용한다.

## 0. 사용자 결정과 최종 목표

사용자는 현재 구현이 인증·보안·테스트·관리 기반을 과도하게 먼저 만들고도 실제 업무의 시작점인 학생 성적 입력과 실제 대학 데이터 연결을 완성하지 못했다고 판단했다. 이 판단을 이 Phase의 최우선 제품 요구사항으로 취급한다.

이번 Phase는 기존 구조를 조금 보완하는 작업이 아니다. 사용자가 제공한 상담 계산 XLSX의 작동 구조를 웹 애플리케이션의 핵심 업무 프로세스로 다시 세우는 전면 재구축이다.

최종 사용자 흐름은 다음과 같아야 한다.

```text
비로그인으로 1회 계산 시작
→ 가상 예시 성적 불러오기 또는 성적 직접 입력·표 붙여넣기·CSV/XLSX 업로드
→ 사용자가 추출값을 화면에서 확인·수정
→ 실제 공개 대학·캠퍼스·학과 선택
→ 전형별 지원자격 독립 판정
→ 지원 가능한 전형만 대학별 반영 평균등급 계산
→ 실제 공개 과거 입시결과와 같은 업무키로 비교
→ 적용 학기·과목·가중치·반올림·출처를 사용자가 확인
→ 학생용·교사용 A4 출력
→ 저장이 필요할 때만 학생 또는 교사 로그인
```

인증은 저장·재조회·학생 상담정보 관리의 경계다. 단순한 1회 성적 계산과 지원 가능 여부 확인의 입구가 아니다.

테스트 수, 모델 수, 문서 수를 진척의 대리 지표로 사용하지 않는다. 실제 브라우저에서 위 흐름이 처음부터 끝까지 동작하는 것이 첫 번째 완료 조건이다.

## 2. 읽기 전용 기준 자료

다음 원본은 Git에 추가하지 않고 현재 위치에서 읽기 전용으로 분석한다.

원본 PDF·XLSX·CSV·노트북 자체, 렌더링 이미지, OCR 파생물, 학생 성적 원문은 커밋하지 않는다. 공개 가능한 대학·학과·입시결과만 출처·해시·기준연도·정규화 상태를 가진 정제 자료로 `data/seed/`에 둘 수 있다.

기준 XLSX의 실행 시작 시 기대값은 다음과 같다. digest가 다르면 파일이 바뀐 것이므로 분석·구현을 시작하지 말고 `BLOCKED_SOURCE`로 기록한다.

```text
size: 394539 bytes
sha256: bde8fe5d513ce2737c08815b0d7e1df366dc8844e6ff7f243eccb63c3bd40606
```

### 2.1 기준 XLSX의 확인된 구조

기준 XLSX는 네 개 시트로 구성된다.

1. `성적 입력`
   - 1학년 1·2학기, 2학년 1·2학기, 3학년 1학기 입력
   - 과목, 단위수, 석차등급
   - 3학년은 원점수, 과목평균, 표준편차로 Z점수와 석차등급 계산
   - 이 시트의 예시 학생 성적은 공개 seed에 넣지 않는다.
2. `대학별 등급`
   - 실제 대학명과 전형별 성적 산출 방식 48행
   - 우수 학기, 학년별 가중치, 이수단위 가중 등 계산 구조
   - `J18`, `J20`처럼 기존에 확인된 참조 불일치·혼합 수식은 그대로 자동 게시하지 않는다.
   - 공식 모집요강과 일치한 산식만 공식 규칙으로 게시한다. 불일치·미확인은 `REFERENCE_ONLY` 또는 `NEEDS_REVIEW`로 보여준다.
3. `2025 수시(1차) 결과`
   - 실제 공개 입시결과 1,818행
   - 지역, 대학명, 모집시기, 전공명, 모집인원, 주야, 전형구분, 합격자 최저·평균 등
4. `2025 수시(2차) 결과`
   - 실제 공개 입시결과 1,652행
   - 1차와 같은 구조

XLSX의 업무 구조는 다음과 같이 웹 프로세스에 직접 대응시킨다.

```text
성적 입력 시트
→ 웹 성적 입력·붙여넣기·업로드·검수 화면

대학별 등급 시트
→ 출처와 버전을 가진 제한형 성적 규칙 + 반영등급 계산 엔진

2025 수시 1차·2차 결과 시트
→ 실제 공개 입시결과 canonical dataset + 비교 화면
```

## 3. 정확히 세 개의 직접 하위 에이전트

메인 에이전트는 시작 즉시 정확히 세 개의 직접 하위 에이전트를 생성한다. `agents.max_depth=1`을 유지하고 하위 에이전트가 다시 에이전트를 생성하지 못하게 한다. 같은 파일을 두 에이전트가 동시에 편집하지 않는다.

### 에이전트 A: 공개 계산 수직 슬라이스 구현

소유 범위:

- 공개·익명 계산 route와 서비스
- 학생 성적 직접 입력·표 붙여넣기·파일 업로드 시작 화면
- 임시 1회 세션과 입력 검수 연결
- 대학·학과 선택, 자격 판정, 계산, 결과, A4 화면
- 로그인 없이 동작하는 핵심 Playwright 흐름
- 관련 템플릿·CSS·Vanilla JavaScript

우선 기존에 고립된 `/input/review/<review_session_id>` 앞에 실제 사용자가 접근할 수 있는 입력 시작 route를 연결한다. 테스트가 임의로 `ReviewState`를 생성하는 방식으로 이 단절을 다시 숨기지 않는다.

### 에이전트 B: 실제 공개 데이터와 연도별 import 구현

소유 범위:

- 기준 XLSX와 CSV/XLSX 입시결과 파일의 읽기 전용 분석
- 공개 입시결과 canonical schema와 연도 전환 계약
- CSV/XLSX 업로드 → 자동 열/시트 탐지 → 정규화 → 오류/중복 미리보기 → 확인 게시
- 실제 대학·캠퍼스·학과·모집시기·전형 업무키 매핑
- 공개 가능한 파생 seed와 출처 manifest
- 관리자용 연도별 입시결과 import 화면과 서비스
- 관련 모델·migration·통합 테스트

원본 자료의 학생 성적 시트는 절대 공개 seed로 내보내지 않는다. `동양미래대학교(특)` 같은 표기는 별도 대학으로 만들지 말고 같은 대학과 전형 구분으로 정규화하되, 확실하지 않은 매핑은 자동 확정하지 않고 검수 항목으로 남긴다.

### 에이전트 C: 독립 업무 흐름 검증

첫 실행은 읽기 전용이다.

- 현재 앱에서 사용자가 시작할 수 없는 흐름과 seed 우회를 문서화한다.
- 요구사항별 수용 매트릭스를 만든다.
- XLSX 원본 수식·캐시값과 앱 결과를 대조할 대표 사례를 정한다.
- 익명/학생/교사/관리자 권한 경계를 검토한다.

에이전트 A와 B의 구현이 끝난 뒤 같은 에이전트에 후속 검증을 맡긴다.

- 전체 diff 독립 리뷰
- 공개 1회 계산 브라우저 E2E
- 실제 데이터 import와 연도 전환 검증
- 입력값을 바꿨을 때 결과가 바뀌는지 검증
- 원본·개인정보·비밀값 비커밋 검사
- PostgreSQL migration·통합·Docker·운영 회귀 검증

검증 에이전트는 기능 코드를 직접 고치지 않고 결함을 메인 에이전트에 보고한다. 메인 에이전트가 담당 구현 에이전트에 후속 수정을 보내고, 같은 검증 에이전트가 재검증한다.

메인 에이전트는 공유 파일 통합, 에이전트 간 경로 충돌 해결, 최종 제품 판정을 책임진다.

## 4. 제품 모드와 인증 경계

### 4.1 비로그인 1회 계산

다음 기능은 로그인 없이 사용할 수 있어야 한다.

- 빈 성적표로 시작
- 합성 예시 성적 불러오기
- 과목별 직접 입력·수정·삭제
- 표 텍스트 붙여넣기
- CSV/XLSX 업로드
- 실제 공개 대학·학과 검색과 복수 선택
- 실제 공개 규칙이 있는 전형의 지원자격 판정
- 대학별 반영 평균등급 계산
- 실제 과거 입시결과 비교
- 계산 근거 확인과 A4 출력

익명 계산에는 이름, 학번, 이메일, 학교명 같은 식별정보를 요구하지 않는다. 내부 학생 코드를 필수 입력으로 요구하지 않는다.

여러 요청에 걸친 입력 검수가 필요하면 짧은 만료시간을 가진 임시 서버 세션을 사용한다. 새 계산, 완료, 만료 시 임시 원본과 파생물을 삭제한다. 계정 DB의 학생 성적 테이블에 익명 입력을 영구 저장하지 않는다.

### 4.2 학생 로그인

학생 로그인은 다음을 위해 존재한다.

- 자신의 성적 저장·수정·삭제
- 관심 대학·학과와 이전 계산 결과 재조회
- 자신의 개인정보와 보유 데이터 관리

학생은 다른 학생이나 교사 상담정보를 볼 수 없다.

### 4.3 교사 로그인

교사 로그인은 다음을 위해 존재한다.

- 여러 학생의 상담자료 관리
- 학생별 메모·상담 이력·출력 저장
- 학생 성적표 업로드와 검수

기존 `MEMBER`를 무조건 관리자 승인 대기 교직원으로만 보는 구조를 재검토한다. 필요한 경우 `STUDENT`, `TEACHER`, `ASSISTANT_ADMIN`, `ADMIN` 역할과 비파괴 migration을 도입한다. 관리자 역할은 공개 계산과 분리한다.

### 4.4 데모 계정과 데모 성적

데모 계정은 유지할 수 있다. 그러나 데모 전용 하드코딩 계산 엔진이 제품의 핵심 계산을 대체해서는 안 된다.

- 데모 성적은 공개 계산 화면에 채워지는 합성 예시 입력이다.
- 사용자가 과목·단위수·등급·원점수·평균·표준편차를 자유롭게 바꿀 수 있어야 한다.
- 실제 공개 대학·학과·입시결과와 동일한 일반 계산 서비스를 호출한다.
- 가상 대학·가상 전형·하드코딩 결과는 공개 제품의 기본 결과에서 제거한다.
- 값 변경 없이 고정 결과만 보여주는 데모는 허용하지 않는다.

공식 규칙이 없는 실제 대학은 가상 규칙으로 채우지 않고 `계산 기준 준비 중`으로 표시한다.

## 5. 학생 성적 입력 화면

기준 XLSX의 `성적 입력` 시트를 사용자에게 이해하기 쉬운 웹 입력 화면으로 구현한다.

최소 입력 필드:

- 학년도
- 학년
- 학기
- 성적 출처(`HOME_SCHOOL_RECORD`, `VOCATIONAL_TRAINING_RECORD`, `GED_RECORD`, `MANUAL_INPUT`)
- 위탁학기 여부
- 교과
- 과목
- 이수단위
- 석차등급
- 원점수
- 과목평균
- 표준편차
- 성취도
- 수강자수
- 필요 시 성취도 분포

요구사항:

- 1·2학년 4개 학기와 3학년 1학기를 기본 그리드로 보여준다.
- 행 추가·삭제는 JavaScript 보조 기능이며 서버 제출은 JavaScript 없이도 동작한다.
- 숫자 0, 빈 값, `P`를 구분한다.
- 표준편차 0이나 누락값으로 Z점수를 임의 생성하지 않는다.
- 입력한 성적을 저장하기 전에도 현재 임시 세션에서 계산할 수 있다.
- 붙여넣기·업로드는 반드시 미리보기와 수정 화면으로 연결한다.
- 검수 완료 후 사용자가 대학 선택 단계로 바로 이동할 수 있다.
- 결과 화면에서 `성적 수정`으로 돌아가 동일 입력을 편집할 수 있다.

## 6. 실제 공개 대학·규칙·입시결과

### 6.1 공개 데이터 원칙

다음을 서로 구분한다.

- 실제 학생 성적·원본 학생 파일: 비공개, Git 금지
- 공개 대학명·캠퍼스·학과·모집시기·전형: 실제 데이터 사용
- 공개 과거 입시결과: 실제 데이터 사용
- 성적 산식·지원자격: 공식 모집요강과 연결된 규칙만 공식으로 표시
- 기준 XLSX 산식: 공식 자료와 일치 여부를 대조하는 참고 계약
- 합성 데이터: 테스트 학생과 예시 입력에만 사용

공개 데이터를 개인정보처럼 숨기지 않는다. 원본 파일을 Git에 넣지 않는 것과 공개 가능한 정제 결과를 앱에서 사용하지 않는 것을 혼동하지 않는다.

### 6.2 공식 규칙 수직 슬라이스

`tmp/codex-reference/pdfs/`의 2027 모집요강을 읽기 전용으로 사용해 최소 다음 실제 대학의 공개 계산 수직 슬라이스를 완성한다.

- 동양미래대학교
- 명지전문대학
- 인하공업전문대학
- 연성대학교
- 가능하면 한양여자대학교, 숭의여자대학교, 배화여자대학교, 경민대학교

각 규칙은 문서명·학년도·페이지·표 위치·버전·검수 상태를 표시한다. Codex가 사람 승인을 위조하지 않는다. 기존 DB의 게시 게이트 때문에 실제 사용이 불가능하다면 사용자 제공 공식 문서에 대한 명시적 `VERIFIED_SOURCE`와 별도 관리자 최종 확인 단계를 설계하되, 가상 규칙으로 대신하지 않는다.

### 6.3 XLSX 비교 가능성

지원하는 대표 입력에 대해 다음을 사용자와 검증자가 확인할 수 있어야 한다.

- XLSX 학기 평균과 앱 학기 평균
- 우수 1개·2개 학기 선택
- 학년별 30/30/40 등 가중치
- 이수단위 가중 평균
- Z점수와 등급 경계
- 반올림 전후 값

앱 결과에는 적용한 규칙, 선택된 과목·학기, 제외 이유, 중간값을 보여준다. 사용자가 어떤 입력 때문에 결과가 달라졌는지 추적할 수 있어야 한다.

## 7. 연도별 CSV/XLSX 입시결과 import

목표 사례:

```text
2027학년도 입시 점수·등급 구간을 크롤링한 CSV/XLSX 업로드
→ 2027 결과 데이터로 보존
→ 기본 상담 대상 학년도를 2028로 제안
→ 관리자가 연도·열 매핑·중복·오류 확인
→ 2028학년도 상담 화면에서 2027 결과를 비교자료로 사용
```

`결과 학년도`와 `상담 대상 학년도`를 별도 필드로 저장한다. 기본값은 `상담 대상 학년도 = 결과 학년도 + 1`로 제안할 수 있지만 화면에서 명시적으로 확인하게 하며, 원본의 학년도를 덮어쓰지 않는다.

전년도 결과와 다음 연도 규칙을 같은 것으로 취급하지 않는다.

- 2028 공식 규칙은 2028 지원자격과 현재 반영등급 계산에 사용한다.
- 2027 역사 규칙이 확인되면 같은 학생 성적을 2027 규칙으로도 별도 계산해 2027 결과와 직접 비교한다.
- 2027 역사 규칙이 없으면 2027 결과 수치는 제공하되 `산식 미확인·참고용`으로 표시한다.
- 현재 규칙으로 계산한 등급을 산식이 다른 과거 결과와 직접 비교하지 않는다.

### 7.1 canonical 필드

최소 필드:

```text
source_code
source_dataset_version
source_row_number
result_academic_year
target_academic_year
region
institution_code
institution_name
campus_code
campus_name
admission_round_code
admission_round_name
program_code
program_name
day_night
admission_category
admission_track_code
admission_track_name
capacity
applicant_count
admitted_count
competition_rate
highest_grade
average_grade
lowest_grade
score_basis
score_direction
historical_score_rule_id
historical_score_rule_version
historical_score_rule_year
source_reference
collected_at
```

선택적 급간 데이터가 있으면 별도 자식 구조로 `band_code`, `lower_bound`, `upper_bound`, 경계 포함 여부, `band_label`을 보존한다. 등급처럼 낮을수록 좋은 척도에서 `highest/lowest`라는 모호한 이름 대신 canonical 의미를 `best_score`, `average_score`, `cutoff_score`와 `score_direction`으로 고정한다.

모든 숫자 필드는 `NULL`과 0을 구분한다. 등급과 배점 척도를 혼합하지 않는다. 원본의 대학명·학과명·전형명이 canonical 코드에 매핑되지 않으면 자동 추정하지 않고 검수 대기 행으로 남긴다.

### 7.2 업로드 처리

```text
CSV/XLSX 업로드
→ 파일 해시와 형식 확인
→ XLSX 시트 자동 탐지
→ 머리글·열 동의어 자동 매핑
→ 결과 학년도·상담 대상 학년도 확인
→ canonical 정규화
→ 대학·캠퍼스·학과·전형 코드 매핑
→ 중복·누락·척도·범위 오류 표시
→ 유효/검토/오류 행 미리보기
→ 관리자 확인
→ versioned published dataset 생성
→ 기존 버전은 삭제하지 않고 supersede
```

기준 XLSX의 수시 1차·2차 시트를 이 import 흐름으로 실제 처리한다. 3,470행을 원본과 행 수·대학 수·평균/최저값 존재 건수로 대조한다. 공개 가능한 정규화 결과는 출처 manifest와 함께 seed 또는 운영 import artifact로 만들 수 있다.

업로드 때마다 개발자가 코드를 수정하거나 대학별 CSV 전용 파서를 새로 만들지 않도록 열 동의어와 매핑 테이블을 데이터로 관리한다. 임의 수식을 실행하지 않으며 CSV formula injection을 차단한다.

### 7.3 게시와 상담 연결

- 데이터셋 단위 상태는 `STAGED`, `READY`, `PUBLISHED`, `SUPERSEDED`, `BLOCKED`를 사용한다.
- 오류 행이 있으면 어떤 행이 왜 막혔는지 보여준다.
- 공개 가능한 유효 행을 명시적으로 게시하되, 부분 게시 정책은 화면에 분명히 표시하고 원본 행 수와 게시 행 수를 보존한다.
- 상담 비교는 `target_academic_year`, 대학, 캠퍼스, 학과, 모집시기, 전형, `score_basis`가 맞는 행만 사용한다.
- 현재 규칙과 과거 규칙의 산식이 다르면 `참고용`으로 표시한다.
- 과거 규칙 버전이 확인된 경우에만 같은 과거 산식으로 학생 성적을 별도 계산해 직접 비교한다.
- 새 연도 데이터셋 게시 후 앱 재배포 없이 상담 화면에서 선택 가능해야 한다.

## 8. 화면 정보 구조

공개 첫 화면의 우선순위:

1. `로그인 없이 성적 계산하기`
2. `예시 성적으로 체험하기`
3. 학생 로그인
4. 교사 로그인
5. 관리자 기능

공개 계산 화면은 다음 단계가 한눈에 보여야 한다.

```text
1. 성적 입력
2. 학생 사실 입력
3. 대학·학과 선택
4. 계산 결과와 근거
```

복수 페이지로 나눌 수 있지만 사용자가 현재 단계, 저장 여부, 이전 단계로 돌아가는 방법을 항상 알 수 있어야 한다.

결과 표의 계층은 `대학 → 학과 → 모집시기 → 전형`이다. 각 행에 다음을 표시한다.

- 지원자격 상태와 이유
- 학생 반영 평균등급
- 반영 학기 또는 핵심 기준
- 공개 입시 평균등급·최저등급·경쟁률·모집인원
- 입시결과 기준연도
- 공식 규칙 또는 참고자료 상태
- 근거 보기

`안정`, `적정`, `소신`, `위험`, `합격 가능성`, `합격 확률`은 생성하지 않는다. 대신 학생 등급과 공개 결과의 숫자 차이를 중립적으로 표시할 수 있다.

## 9. 기존 구현 정리

기존 파일을 무조건 보존하는 것이 목표가 아니다. 다음 기준으로 판단한다.

- 실제 사용자 흐름을 제공하는 기존 도메인 엔진·모델은 재사용한다.
- 고립된 검수 route는 공개 입력 시작점과 연결한다.
- `demo_consultations.py`의 하드코딩 가상 대학·규칙·성적·결과가 일반 계산을 우회하면 제거하거나 단순 예시 입력 fixture로 축소한다.
- DB seed를 직접 주입해야만 통과하는 E2E는 공개 사용자 진입점 E2E로 교체한다.
- 더 이상 존재하지 않는 동작을 고정하는 테스트는 삭제만 하지 말고 새 업무 요구를 검증하는 테스트로 대체한다.
- 제품 가치 없이 복잡도만 늘리는 wrapper·중복 서비스·죽은 route·죽은 템플릿은 참조 확인 후 정리할 수 있다.
- 보안·개인정보 보호는 유지하되 공개 계산 자체를 인증으로 막지 않는다.

대규모 리팩터링은 한 번에 모든 파일을 바꾸는 것으로 정의하지 않는다. 먼저 공개 수직 슬라이스를 완성하고, 그 흐름에 연결되지 않는 구조를 증거에 따라 제거한다.

## 10. 테스트 원칙: 업무 흐름 우선

테스트를 위한 테스트를 만들지 않는다.

- 테스트 개수나 커버리지 증가를 완료 조건으로 보고하지 않는다.
- 한 개의 실제 사용자 수직 슬라이스를 검증하는 소수의 고가치 테스트를 우선한다.
- 내부 helper만 검증하면서 사용자 진입점 부재를 숨기지 않는다.
- 테스트용 DB seed는 import UI나 공개 입력 UI를 검증하는 테스트를 대체할 수 없다.
- HTTP 상태 코드만 확인하지 않는다.
- 사용자 입력값을 바꾸면 결과와 trace가 실제로 바뀌는지 확인한다.
- 실제 공개 데이터의 행 수·대학 수·업무키와 source manifest를 대조한다.
- 테스트 fixture의 합성 학생 자료는 허용하지만 가상 대학 결과로 실제 데이터 연결을 대신하지 않는다.

기준 문서의 TDD 원칙은 다음 방식으로 적용한다.

```text
한 개의 핵심 사용자 실패 E2E 또는 수용 테스트
→ 실패 이유 확인
→ 최소 수직 구현
→ 실제 브라우저 수동·자동 확인
→ 필요한 단위·통합 회귀만 추가
```

## 11. 필수 수용 기준

### 공개 계산

1. 로그아웃 상태에서 공개 계산 시작 화면을 열 수 있다.
2. 내부 학생 코드·계정 없이 성적을 직접 입력할 수 있다.
3. 합성 예시 성적을 불러온 뒤 임의 과목 등급을 바꿀 수 있다.
4. CSV/XLSX 또는 표 붙여넣기로 성적 입력 검수 세션을 시작할 수 있다.
5. 검수 완료 뒤 대학 선택 단계로 자연스럽게 이동한다.
6. 실제 대학명·학과명·모집시기·전형이 표시된다.
7. 공식 규칙이 있는 전형은 자격을 먼저 판정한 뒤에만 계산한다.
8. 성적 한 값을 바꾸면 해당 대학 반영등급과 trace가 예상 방향으로 바뀐다.
9. 공식 규칙이 없는 대학은 가상 결과 대신 `계산 기준 준비 중`으로 표시된다.
10. 결과에 실제 과거 입시 평균·최저·경쟁률·기준연도가 표시된다.
11. 학생용·교사용 A4가 로그인 없이 현재 1회 결과를 출력한다.
12. 새 계산 또는 만료 후 익명 임시 입력과 업로드 원본이 삭제된다.

### 엑셀 재현·검증

13. 기준 XLSX의 지원 가능한 대표 수식과 앱 결과가 같은 입력에서 일치한다.
14. `J18`, `J20`의 알려진 불일치는 자동 공식 규칙으로 게시되지 않는다.
15. 계산 trace만으로 선택 학기·과목·가중치·반올림을 재현할 수 있다.

### 실제 데이터 import

16. 기준 XLSX의 수시 1차 1,818행과 수시 2차 1,652행을 import 흐름으로 읽는다.
17. 원본 학생 성적 시트의 값은 공개 seed·로그·DB 입시결과에 들어가지 않는다.
18. CSV와 XLSX 모두 동일 canonical 결과를 만들 수 있다.
19. 2027 결과 파일을 올리면 결과연도 2027, 기본 상담 대상연도 2028로 미리보기된다.
20. 관리자가 대상연도를 확인·수정할 수 있다.
21. 새 데이터셋 게시 후 코드 수정·재배포 없이 해당 상담연도에서 조회된다.
22. 중복·빈 업무키·알 수 없는 대학/학과/전형·다른 점수 척도는 명확히 검수 대기 또는 차단된다.
23. 같은 데이터셋 재업로드는 해시 중복으로 식별된다.
24. 새 버전 게시 시 이전 버전은 삭제되지 않고 `SUPERSEDED`로 남는다.

### 인증과 저장

25. 익명 계산에는 로그인이 필요 없다.
26. 저장 버튼을 누를 때만 학생·교사 로그인을 안내한다.
27. 학생은 자기 저장 자료만 조회·삭제할 수 있다.
28. 교사는 허용된 학생 상담자료와 메모를 관리할 수 있다.
29. 관리자는 공개 데이터 import와 규칙 검수를 수행한다.
30. 데모 계정은 실제 관리자 권한을 얻지 않으며, 데모 입력도 일반 계산 엔진을 사용한다.

### 사용자 경험과 회귀

31. JavaScript를 끈 상태에서도 직접 입력→대학 선택→계산 제출이 가능하다.
32. 모바일 390px에서 입력·결과 핵심 항목이 잘리지 않는다.
33. 브라우저 console error와 page error가 0건이다.
34. 실제 개인자료·원본 자료·비밀값이 Git diff에 없다.
35. 기존 PostgreSQL volume과 사용자 계정을 보존하는 migration을 사용한다.

## 12. 구현 게이트

### Gate A: 공개 세로 흐름

다음이 실제 브라우저에서 동작하기 전에는 AI, 추가 보안 기능, 테스트 수 확대, 관리자 편의 기능을 새로 만들지 않는다.

```text
로그아웃
→ 예시 성적 불러오기
→ 성적 한 값 수정
→ 실제 대학·학과 선택
→ 자격 판정
→ 반영 평균등급과 실제 입시결과 확인
→ 근거와 A4 확인
```

### Gate B: 실제 공개 데이터

기준 XLSX의 실제 입시결과를 canonical import로 처리하고, 공개 상담 화면에서 기준연도를 포함해 조회한다.

### Gate C: 연도 갱신

새 연도 CSV/XLSX를 관리자 화면에서 업로드하고 다음 상담연도에 게시한다. 코드 수정이나 Docker 재빌드 없이 데이터만 갱신되는 것을 확인한다.

### Gate D: 로그인 저장

익명 계산과 로그인 저장 경계를 확인하고 학생·교사 소유권을 검증한다.

### Gate E: 회귀·운영

핵심 사용자 E2E, 관련 단위·통합, 정적 검사, 민감자료 검사, PostgreSQL migration, Docker health를 통과한다.

각 Gate를 통과할 때 사용자가 수행한 행동, 화면 결과, 저장 데이터 또는 삭제 상태를 기록한다. 테스트 건수만 적지 않는다.

## 16. Git·push·Docker 권한과 완료 순서

사용자는 Phase 14에 한해 다음을 명시적으로 승인했다.

- 저장소 내부 파일 수정·생성·정리
- 테스트·정적 검사·Playwright·PostgreSQL 통합 실행
- 이 프로젝트의 Docker 이미지 빌드·재빌드
- 이 프로젝트의 애플리케이션 컨테이너 재생성·재시작
- 필요한 비파괴 migration
- Phase 14 변경의 한국어 Git 커밋
- 현재 브랜치의 정상 원격 push

중간 승인을 반복 요청하지 말고 안전한 범위에서 완료까지 진행한다.

### Git

- 시작 시 dirty 변경을 기록하고 사용자 변경을 보존한다.
- `git add -A`, `git commit -a`를 사용하지 않는다.
- Phase 14에서 실제 수정한 경로만 명시적으로 stage한다.
- `git diff`, `git diff --cached`, 민감자료 검사를 커밋 전에 확인한다.
- 커밋 메시지는 자연스러운 한국어로 작성한다.
- 현재 브랜치를 일반 push한다. force push하지 않는다.
- push 후 로컬 HEAD와 원격 SHA를 확인한다.

### Docker와 운영 데이터

- 먼저 `docker compose ls`, `docker ps`, 운영 문서로 이 프로젝트의 실제 Compose 프로젝트·파일·서비스를 식별한다.
- 다른 프로젝트를 건드리지 않는다.
- 기존 PostgreSQL volume과 계정·상담 데이터를 보존한다.
- migration이 필요하면 운영 백업과 복원 가능성 검증을 먼저 수행한다.
- 테스트와 민감자료 검사가 통과한 커밋만 빌드한다.
- 전체 stack을 내리지 말고 필요한 애플리케이션 서비스만 빌드·재생성한다.
- DB 컨테이너는 불필요하게 재생성하지 않는다.
- 새 이미지의 migration head, health, loopback origin, 공인 HTTPS, 핵심 공개 계산·관리자 import smoke를 확인한다.
- 실패하면 성공으로 보고하지 말고 기존 이미지와 보존된 DB로 안전하게 복구한다.

### 완료 순서

1. Gate A~E 구현과 독립 검증
2. 전체 diff·민감자료·원본 포함 여부 확인
3. Phase 14 구현 한국어 커밋
4. 현재 브랜치 원격 push
5. 운영 백업·필요 migration·애플리케이션 Docker 재빌드/재시작
6. health와 실제 공개 사용자 흐름 smoke
7. 배포 기록 문서가 바뀌면 별도 한국어 기록 커밋과 push
8. 로컬·원격 SHA, Docker image/container, DB migration, 공개 URL 상태 확인

네트워크·Git 인증·Docker daemon·외부 환경이 실제로 막히면 무한 반복하지 않는다. 안전한 대안을 소진한 뒤 정확한 명령·오류·마지막 성공 단계·미완료 범위를 최종 보고한다.
