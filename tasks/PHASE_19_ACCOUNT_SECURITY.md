# Phase 19 작업 카드 — 이메일 로그인과 계정 보안

## 목표

- 신규 학생·교사 가입은 검증 가능한 이메일과 비밀번호를 사용한다.
- 기존 아이디 계정은 중단 없이 `이메일 또는 기존 아이디`로 로그인한다.
- 이메일 소유 확인과 관리자 역할 승인을 별도 게이트로 적용한다.
- 사용자는 계정 보안 화면에서 비밀번호를 변경하고 모든 기존 세션을 무효화한다.
- 검증된 이메일로 일회용·만료형 비밀번호 재설정을 수행한다.
- 로그인한 사용자가 재인증 뒤 Google `(issuer, subject)`를 명시적으로 연결·해제한다.
- 기존 아이디 계정은 계정 보안 화면에서 실제 이메일을 변경·검증한 뒤 이메일 로그인과 복구를 사용할 수 있다.
- 동일 이메일만으로 계정 또는 학생·성적·상담 데이터를 자동 병합하지 않는다.

## 선행조건과 근거

- Phase 18 운영 계정·역할·BYOK 배포가 완료되어 있어야 한다.
- `2027_전문대_입시상담앱_실행개발문서_v2.md`의 회원가입·개인정보 원칙을 유지한다.
- `CODEX_MASTER_PROMPT_2027_전문대_입시상담앱_v3.md`의 단계 게이트와 운영 백업 절차를 따른다.
- 이메일 인증은 계정의 주소 소유 확인이며 학생·교사 역할 승인을 대신하지 않는다.

- DB migration: `migrations/versions/`
- 합성 익명 검증: `tests/`, `e2e/`
- 운영 설정·자동화: `.env.example`, `docker-compose*.yml`, `scripts/`, `Makefile`
- 상태·운영 문서: `README.md`, `DEVELOPMENT_LOG.md`, `PROJECT_STATUS.md`, 이 작업 카드

## 금지 사항

- 실제 이메일 주소, SMTP 비밀번호, Google client secret, reset/verification 원문 token을 Git·DB·로그에 기록
- 이메일 일치만으로 로컬 계정과 Google 계정을 자동 연결 또는 데이터 병합
- 검증 메일 성공을 학생·교사 계정 승인으로 간주
- 데모 계정의 비밀번호·이메일·Google 연결 변경 허용
- 기존 아이디 계정을 검증 이메일 등록 전에 강제로 잠금
- 사용자 미추적 파일이나 두 기준 원본 문서 변경

## Given / When / Then 수용 기준

1. 신규 가입자가 이메일·비밀번호를 제출하면 검증 전·승인 대기 계정과 digest만 저장된 만료 token이 생성된다.
2. 유효한 이메일 인증 token을 한 번 사용하면 이메일만 검증되며 계정은 관리자 승인 전까지 업무 화면에 들어갈 수 없다.
3. 로그인은 검증 이메일과 기존 아이디를 모두 허용하고, 존재하지 않는 계정 여부를 오류 문구로 노출하지 않는다.
4. 검증된 이메일의 비밀번호 찾기는 동일한 접수 응답을 반환하고 일회용·만료형 token으로 비밀번호를 변경한다.
5. 활성 비데모 사용자가 현재 비밀번호를 재확인해 변경하면 다른 세션은 무효화되고 현재 세션만 새 버전으로 갱신된다.
6. 로그인한 로컬 사용자가 현재 비밀번호 재확인과 Google OIDC 검증을 모두 통과하면 같은 계정에 `(issuer, subject)`가 연결된다.
7. Google 로그인에서 동일 이메일의 기존 계정을 발견해도 자동 연결하지 않고 기존 계정 로그인 후 연결하도록 안내한다.
8. Google 연결 해제는 최근 재인증과 대체 비밀번호 로그인 수단이 있을 때만 허용한다.
9. 데모 계정은 계정 보안 변경 요청이 403으로 차단된다.
10. SMTP·Google 외부 자격증명이 없는 배포는 해당 기능의 구성 필요 상태를 명확히 표시하며 기존 로그인은 유지한다.
11. 기존 `@local.invalid` 계정은 기존 아이디 로그인을 유지하고 실제 이메일을 검증하기 전에는 이메일 로그인·복구 대상으로 사용하지 않는다.
