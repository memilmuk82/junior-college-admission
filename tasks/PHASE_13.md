# Phase 13 작업 카드 겸 Codex 실행 프롬프트

이 파일 전체를 Codex CLI 비대화형 실행의 프롬프트로 사용한다.

## 최종 목표

가상 학생의 성적과 학생 사실을 한 번 입력한 뒤 사용자가 여러 대학의 여러 희망학과를 체크하면, 각 학과에 연결된 모든 게시 전형의 지원자격을 독립 판정하고 지원 가능한 전형의 대학별 반영 평균등급을 한 화면과 A4 출력에서 확인할 수 있게 한다.

핵심 처리 단위는 다음과 같다.

```text
학생 1명
→ 희망 대학·학과 N개 선택
→ 학과별 전형 N개 확장
→ 전형별 지원자격 독립 판정
→ 지원 가능한 전형별 반영 평균등급 계산
→ 학과별 공개 입시 평균등급 연결
→ 전체 결과 표시
```

사용자에게 보여줄 핵심 성적은 배점 환산점수가 아니라 `대학별 반영 평균등급`이다.

## 기준 문서와 시작 절차

코드를 수정하기 전에 다음을 수행한다.

1. `AGENTS.md`를 읽고 저장소 지침을 준수한다.
2. 다음 기준·현황 문서를 읽는다.
   - `2027_전문대_입시상담앱_실행개발문서_v2.md`
   - `CODEX_MASTER_PROMPT_2027_전문대_입시상담앱_v3.md`
   - `PROJECT_STATUS.md`
   - `DEVELOPMENT_LOG.md`
   - `tasks/PHASE_12.md`
   - 관련 `docs/` 문서
3. 기준 원본 두 문서는 수정·삭제·이동·축약·덮어쓰지 않는다.
4. `git status --short`, `git diff`, `git diff --cached`를 확인한다.
5. 시작 시점에 존재하는 변경은 사용자 소유로 간주하고 보존한다.

프롬프트 준비 과정에서 다음 파일에 별도 Phase 12 변경이 존재한 이력이 있다. 실제 실행 시작 시 `git status`와 staged diff를 다시 확인하고, 해당 변경이 남아 있다면 수정·되돌림·unstage하거나 Phase 13 커밋에 섞지 않는다.

- `DEVELOPMENT_LOG.md`
- `PROJECT_STATUS.md`
- `tasks/PHASE_12.md`

예외적으로 이 파일과 실행 스크립트는 사용자가 Phase 13을 위해 명시적으로 추가한 작업 자산이므로 Phase 13 최종 커밋에 포함한다.

- `tasks/PHASE_13.md`
- `scripts/run_phase13_codex_background.sh`

## 에이전트 병렬 운영

메인 에이전트는 구현을 혼자 순차 수행하지 말고 정확히 두 개의 직접 하위 에이전트를 즉시 백그라운드로 생성한다. `agents.max_depth=1` 경계를 유지하며 하위 에이전트가 다시 에이전트를 생성하지 않게 한다.

### 에이전트 1: 코드 조사·구현

역할:

- 현재 상담, 학생 성적, 성적 범위, 계산, 입시결과, 데모, 출력, AI payload 구조를 확인한다.
- 아래 확정 요구사항을 구현한다.
- 변경에 필요한 단위·통합·라우트·E2E 테스트를 함께 작성한다.
- 관련 없는 코드와 사용자 변경을 건드리지 않는다.
- 구현이 끝나면 변경 파일, 설계 결정, 테스트 결과와 남은 위험을 메인 에이전트에 요약한다.

### 에이전트 2: 독립 검증

에이전트 1과 동시에 시작한다. 첫 번째 실행은 읽기 전용 검증으로 제한한다.

1차 역할:

- 기존 코드와 테스트를 독립 분석한다.
- 요구사항별 수용 기준과 검증 매트릭스를 작성한다.
- 현재 기준 테스트를 가능한 범위에서 실행한다.
- 회귀 위험, 데이터 모델 위험, 누락 가능성이 큰 테스트를 보고한다.
- 에이전트 1이 수정 중인 기능 파일을 편집하지 않는다.

에이전트 1이 구현을 마치면 메인 에이전트는 같은 검증 에이전트에 후속 작업을 전달한다.

2차 역할:

- 전체 Phase 13 diff를 독립 리뷰한다.
- 요구사항 누락, 오류, 보안·개인정보 회귀, 테스트 공백을 찾는다.
- 표적 테스트, 전체 단위 테스트, PostgreSQL 통합 테스트, Playwright E2E, 정적 검사와 민감자료 검사를 실행한다.
- 기능 코드를 직접 고치지 말고 결함을 파일과 근거를 포함해 메인 에이전트에 보고한다.

검증에서 결함이 발견되면 에이전트 1에 후속 수정 작업을 전달하고, 수정 후 검증 에이전트가 다시 검증한다. 필수 검증이 통과하거나 외부 환경의 재현 가능한 차단 사유가 확정될 때까지 이 과정을 반복한다.

두 에이전트가 동시에 같은 파일을 수정하게 하지 않는다. 메인 에이전트는 요구사항, 작업 경계, 에이전트 조정, 최종 통합 판정에 집중한다.

## 확정 기능 요구사항

### 1. 학교·학과 복수 선택

- 실제 선택 단위는 대학이 아니라 `대학·캠퍼스·학과(program)`다.
- 폼에서 복수 `program_ids`를 제출한다.
- 대학 아래 학과를 계층형 체크박스로 표시한다.
- 대학 체크박스는 해당 대학 학과 전체 선택·해제를 돕는 점진적 향상 기능이다.
- 학과별 개별 선택과 서로 다른 계열 학과의 혼합 선택을 모두 허용한다.
- 대학명·학과명 검색, 검색 결과 전체 선택, 선택 개수 표시를 제공한다.
- 핵심 학과 선택과 제출은 JavaScript 없이도 동작해야 한다.
- Vanilla JavaScript는 검색·전체 선택·개수 표시를 개선하는 용도로만 사용한다.
- 모집학년도를 명시하고 2027학년도를 기본값으로 한다.
- 선택된 학과가 없거나 허용되지 않은 ID가 포함되면 명확한 검증 오류를 표시한다.
- 중복 ID는 결정적으로 정규화한다.

### 2. 학과별 전형 자동 확장

사용자는 대학·학과까지만 선택한다. 서버는 선택한 각 학과에 연결된 해당 학년도의 전형을 자동 조회한다.

- 전형마다 지원자격을 독립 판정한다.
- `ELIGIBLE`과 `CONDITIONALLY_ELIGIBLE`만 평균등급 계산으로 진행한다.
- `INELIGIBLE`은 성적을 조회·계산하지 않고 불가 사유를 표시한다.
- `NEEDS_REVIEW`와 `INSUFFICIENT_DATA`도 숨기지 않고 검토 상태로 표시한다.
- 게시·검수된 규칙이 없는 대학·학과는 조용히 누락하지 않고 `계산 기준 준비 중`으로 표시한다.
- 한 학과나 전형의 실패가 다른 결과를 제거하지 않도록 항목 단위로 격리한다.
- 복수지원 가능 여부는 전형 지원자격과 별도 상태로 다룬다.

대표 수용 사례:

- 동양미래대학교와 같은 구조에서 동일 학과의 일반고전형과 특성화고전형을 모두 독립 판정한다.
- 같은 학생이 두 전형 모두 지원 가능하면 두 결과를 모두 표시한다.
- 일반고전형은 위탁학기 제외, 특성화고전형은 위탁성적 포함처럼 범위가 다르면 평균등급을 각각 계산한다.
- 실제 대학의 검수 완료 seed가 없으면 실제 규칙을 추정하지 않고 동일 구조의 명시적 합성 데이터로 테스트한다.

### 3. 반영 평균등급

상담 화면의 핵심 결과를 `환산점수 / 만점`에서 `대학별 반영 평균등급`으로 바꾼다.

- 기존 trace의 `aggregate_value`를 템플릿에서 임의로 꺼내 쓰는 임시 구현으로 끝내지 않는다.
- `AverageGradeResult`, `ReflectedGradeResult`처럼 평균등급 의미를 명시한 결과 모델과 계산 경계를 만든다.
- 결과는 최소한 다음 정보를 보존한다.
  - 반올림 전 평균등급
  - 최종 표시 평균등급
  - 등급 척도
  - 반영 학기와 과목
  - 이수단위 및 학년·학기 가중치
  - 반올림 과정
  - 규칙 ID·버전과 근거
- 우수학기, 학년별 가중치, 이수단위 가중, 위탁학기 포함·제외를 적용한다.
- 진로선택은 공식 변환 규칙과 버전 고정 표가 있을 때만 반영한다.
- 배점 환산 엔진이 공식 규칙 검증에 필요하면 내부 기능으로 유지할 수 있지만 상담의 핵심 결과에는 노출하지 않는다.
- 등급과 점수 척도를 혼합하지 않는다.

### 4. 결과 화면

결과 계층은 `대학 → 학과 → 전형`이다.

요약표에는 다음을 표시한다.

- 대학·캠퍼스
- 학과
- 모집시기
- 전형
- 지원자격
- 학생 반영 평균등급
- 반영 학기 또는 핵심 반영 기준
- 공개 입시 평균등급
- 입시결과 기준연도
- 경쟁률
- 검수·확인 필요 상태

동일 대학의 여러 학과가 같은 성적 규칙을 사용하면 내부 계산을 재사용할 수 있지만, 학과별 입시 평균등급과 경쟁률은 별도 행으로 표시한다. 동일 학과의 전형별 규칙이나 성적 범위가 다르면 전형별 평균등급을 따로 표시한다.

기본 정렬은 사용자 선택 순서 또는 대학·학과의 결정적 표시 순서로 한다. 평균등급순 정렬을 기본값으로 삼거나 이를 추천 순위처럼 표현하지 않는다.

### 5. 공개 입시결과 비교

- 학생 평균등급과 같은 등급 척도임이 검증된 자료만 직접 비교한다.
- 대학·캠퍼스·학과·모집시기·전형·연도 업무키를 정확히 맞춘다.
- 기준연도를 반드시 표시한다.
- 현재 규칙과 과거 규칙이 다르면 `참고용`으로 표시한다.
- `score_basis`가 배점이나 다른 척도이면 직접 비교하지 않는다.
- 공개 평균등급이 없으면 `자료 없음`으로 표시한다.
- 최고·최저 자료가 있어도 핵심 비교값은 공개 평균등급이다.
- 공개 가능하고 사람 검수·승인된 `data/seed/` 자료만 실제 대학 결과로 사용한다.

### 6. 공개 데모

현재 한 개 가상 대학·전형 데모를 복수 대학·학과 선택 흐름으로 확장한다.

- 가상 학생 성적은 한 번만 준비한다.
- 여러 가상 대학과 여러 학과를 선택할 수 있게 한다.
- 전체 학기, 최우수 학기, 학년 가중, 위탁학기 제외·포함 등 서로 다른 평균등급 반영 방식을 포함한다.
- 동일 학과에 일반고·특성화고전형이 모두 연결된 합성 사례를 포함한다.
- 모든 학생·대학·학과·전형·입시결과가 합성임을 화면과 출력에 명확히 표시한다.
- 승인된 실제 대학 공개 seed가 없다면 실제 대학명과 규칙을 데모에 넣지 않는다.

### 7. A4 출력과 AI 계약

- 학생용 A4에는 선택한 모든 대학·학과·전형의 요약 결과를 표시한다.
- 교사용 A4에는 전형별 자격, 반영 학기·과목·가중치·반올림·근거 trace를 추가한다.
- 기존 학생용·교사용 print CSS 분리를 유지한다.
- AI payload가 단일 결과와 점수 필드에 의존하면 명시적 schema version을 올리고 복수 결과와 평균등급을 표현한다.
- AI는 검증된 결과를 문장화할 뿐 자격·등급·추천·순위·합격 가능성을 새로 생성하지 못한다.
- 데모 계정의 AI 접근 차단을 유지한다.

## 권장 구현 구조

기존 단일 전형 판정 함수는 가능한 한 보존하고 그 위에 배치 계층을 추가한다.

- 복수 `program_ids`와 모집학년도를 받는 요청 모델
- 선택 학과를 해당 연도 전형 목록으로 확장하는 조회 서비스
- 기존 단일 전형 판정을 호출하는 항목 단위 실행
- 전체 항목 결과와 항목별 오류·준비 상태를 보존하는 배치 결과
- 점수 결과와 의미적으로 분리된 평균등급 결과
- 동일 규칙·입력 조합에 대한 안전한 계산 재사용
- 저장할 필요가 없는 희망학과 선택 때문에 불필요한 DB migration을 만들지 않음

데이터 모델 변경이 불가피하다고 판단하면 먼저 기존 모델로 표현할 수 없는 이유를 검증 에이전트에게 확인받고 새 migration은 하나로 제한한다.

## 필수 수용 테스트

1. 한 학생이 여러 대학·학과를 동시에 선택할 수 있다.
2. 복수 `program_ids`가 안전하게 파싱되고 빈 선택은 거부된다.
3. 선택 학과별 게시 전형이 빠짐없이 확장된다.
4. 같은 학과의 일반고·특성화고전형이 독립 판정된다.
5. 두 전형 모두 가능하면 두 평균등급이 표시된다.
6. 전형별 성적 범위가 다르면 서로 다른 평균등급이 나온다.
7. 자격 미달 전형은 지원자격 판정 후 성적을 조회·계산하지 않는다.
8. 한 항목의 실패가 다른 대학·학과 결과를 제거하지 않는다.
9. 우수학기·학년 가중·이수단위·위탁학기 포함·제외가 적용된다.
10. 상담 결과와 A4 핵심 영역에 `환산점수 / 만점`이 나타나지 않는다.
11. 학과별 공개 평균등급과 경쟁률이 정확한 업무키로 연결된다.
12. 다른 척도 또는 다른 규칙 연도는 직접 비교되지 않는다.
13. JavaScript 없이 복수 학과 선택과 제출이 가능하다.
14. 학생용·교사용 A4가 모든 선택 결과를 포함한다.
15. 데모가 복수 대학·학과·전형 흐름을 합성 자료로 통과한다.
16. AI payload에 학생 식별정보가 포함되지 않고 복수 평균등급 계약이 고정된다.
17. 실제 학생 자료나 원본 자료가 저장소에 추가되지 않는다.

Playwright E2E는 HTTP 상태만 확인하지 않는다. 실제 체크박스 선택, 복수 학과 제출, 전형별 결과 행, 평균등급, 자격 미달·준비 중 상태, 학생용·교사용 인쇄 화면을 검증한다.

## 검증 원칙

저장소의 Makefile과 기존 테스트 전략을 우선 사용한다. 최소한 다음을 실행한다.

- Phase 13 관련 단위 테스트
- 상담·성적·입시결과 전체 단위 테스트
- PostgreSQL 통합 테스트
- 상담 라우트와 인증 경계 테스트
- Playwright E2E
- Ruff와 포맷 검사
- mypy
- 규칙 seed 검증
- `python scripts/check_sensitive_data.py`
- 가능한 전체 회귀 테스트

테스트 실패를 무시하거나 성공으로 주장하지 않는다. 환경·권한·외부 서비스 문제로 실행할 수 없는 검증은 명령, 오류, 마지막 성공 단계와 미검증 범위를 보고한다.

## Git·Docker·원격 push 권한

사용자는 이 저장소의 Phase 13 작업에 한해 다음 권한을 명시적으로 부여했다.

- 저장소 내부 구현 파일 수정·생성
- 테스트와 정적 검사 실행
- 이 프로젝트의 Docker 이미지 빌드·재빌드
- 이 프로젝트의 Compose 컨테이너 재생성·재시작
- 필요한 비파괴적 DB migration 실행
- Phase 13 변경의 로컬 Git 커밋
- 현재 작업 브랜치의 원격 저장소 일반 push

중간 승인을 사용자에게 요청하지 말고 백그라운드에서 완료까지 진행한다.

### Git 처리

- 커밋 직전에 `git status`, `git diff`, `git diff --cached`를 다시 확인한다.
- 시작 전에 존재하던 사용자 변경을 Phase 13 커밋에 포함하지 않는다.
- `git add -A`나 `git commit -a`처럼 범위가 넓은 명령을 사용하지 않는다.
- Phase 13에서 실제로 변경한 경로만 명시적으로 stage하고 path별 cached diff를 확인한다.
- 이미 stage된 사용자 변경이 있으면 `git commit --only`와 명시적 Phase 13 경로 등 안전한 방식으로 분리한다.
- 커밋 메시지는 자연스러운 한국어로 작성한다.
- force push, reset, clean, rebase, amend, 기존 브랜치·태그 삭제를 하지 않는다.
- 현재 브랜치, upstream, origin URL과 ahead/behind 상태를 확인한다.
- 검증과 컨테이너 health check가 끝난 뒤 현재 브랜치를 정상 push한다.
- upstream이 없으면 현재 저장소의 기존 `origin`을 확인한 뒤 해당 현재 브랜치에 `git push -u`할 수 있다.
- push 후 로컬 HEAD와 원격 브랜치 SHA를 확인한다.

### Docker 처리

- `docker compose ls`, `docker ps`와 저장소 운영 문서를 통해 이 저장소의 실제 활성 Compose 프로젝트·파일·서비스를 먼저 식별한다.
- 다른 프로젝트의 컨테이너·이미지·네트워크·volume에는 손대지 않는다.
- 기존 PostgreSQL volume을 보존한다.
- DB 구조 변경이 없으면 migration이나 DB 재생성을 하지 않는다.
- migration이 필요하면 저장소의 production preflight와 백업·검증 절차를 먼저 수행한다.
- 테스트와 민감자료 검사가 통과한 커밋만 이미지로 빌드한다.
- 전체 stack을 내리지 말고 필요한 애플리케이션 서비스만 기존 Compose 구성으로 빌드·재생성한다.
- DB 컨테이너를 불필요하게 재생성하지 않는다.
- 재빌드 후 컨테이너 상태, health check, 핵심 HTTP 화면과 최근 오류 로그를 확인한다.
- 새 컨테이너가 정상 상태가 아니면 성공으로 보고하지 말고 안전한 복구를 시도한다.
- `down -v`, volume 삭제, DB 초기화, system prune은 금지한다.

Git 또는 Docker 인증·daemon·외부 네트워크가 실제로 사용할 수 없는 경우에는 무한 대기하지 않는다. 안전한 대안을 소진한 뒤 정확한 오류와 마지막 성공 단계까지 최종 보고한다.

## 최종 완료 보고

최종 답변에는 다음을 구체적으로 기록한다.

- 구현된 사용자 흐름
- 주요 설계 결정과 변경 파일
- 구현 에이전트와 검증 에이전트의 수행 내용
- 실행한 테스트·검사 명령과 결과 건수
- Docker Compose 프로젝트·재빌드 서비스·health 결과
- 생성한 한국어 커밋 메시지와 SHA
- push한 원격·브랜치와 원격 SHA
- 보존한 기존 사용자 변경

필수 기능, 독립 검증, 컨테이너 health, 민감자료 검사, 커밋과 push가 모두 끝나기 전에는 완료로 선언하지 않는다.
