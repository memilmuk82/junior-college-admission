# Phase 10 비운영 준비 패키지

이 문서는 운영 환경을 변경하지 않고 다음 다섯 작업을 준비·리허설하기 위한 문서다. 실제 키·학생자료·원본 입시자료·운영 DB는 사용하지 않는다.

## 1. 공통 승인 체크리스트

| 게이트 | 확인 내용 | 승인 기록 |
|---|---|---|
| 범위 | 합성 데이터·격리 DB·임시 secret만 사용 | 담당자/시각/티켓 |
| 백업 | 실행 전 백업과 SHA-256 확인 | 담당자/시각 |
| 권한 | 최소권한 계정·변경창·롤백 담당 확인 | 운영 담당자 |
| 검증 | 단위·통합·E2E·민감자료 검사 계획 확인 | QA 담당자 |
| 종료 | 로그·건수·회귀 결과와 잔여 위험 기록 | 승인자 |

## 2. 상세 런북

### A. 모집요강 버전 교체 리허설

- 목적/범위: `synthetic-` 전형의 기존 PUBLISHED와 신규 DRAFT를 교체하고 영향도를 비교한다.
- 사전 조건: 공개 가능한 합성 seed, 공식 근거를 주장하지 않는 합성 citation, 격리 PostgreSQL.
- 최소 권한: 테스트 DB 쓰기·읽기, 합성 관리자 계정; 운영 DB·배포 권한 금지.
- 사전 점검: 현재 버전·lineage·게시 규칙 수를 조회하고 기존 payload digest를 기록한다.
- 승인 게이트: 독립 검증·골든 테스트·사람 승인 확인 후 게시 단계 진입.
- 절차 초안: 기존 규칙 복제→DRAFT→TESTED→HUMAN_APPROVED→PUBLISHED→기존 SUPERSEDED.
- 검증: 지원자격 차단 후 성적 계산 미호출, 변경 전후 Decimal 영향도, 학생용·교사용 SSR/A4, 저장된 lineage·audit event.
- 실패 기준: 중복 PUBLISHED, 근거 누락, 브라우저 오류, 이전 버전 복원 불가.
- 롤백: 합성 신규 버전을 SUPERSEDED로 하고 기존 합성 버전을 재게시한다.
- 완료 증거: diff, 영향도, 테스트 결과, audit event, 화면·저장 데이터 확인.
- 금지: 실제 대학명·학생·PDF·엑셀·운영 게시 전환.
- 서명: 검수자/승인자/시각/티켓.

합성 회귀 결과:

- 대학 시행계획은 `UNIVERSITY_OFFICIAL / IMPLEMENTATION_PLAN`인 공식 선행 자료로 시작한다.
- 최종 모집요강은 기존 게시 규칙을 덮어쓰지 않고 `FINAL_GUIDE` 근거의 새 DRAFT 규칙 버전으로 생성한다.
- 동일 합성 입력의 환산점수가 `80.00`에서 `64.00`으로 바뀌는 `-16.00` 영향도를 Decimal로 비교했다.
- 검수·테스트·명시적 사람 승인 뒤 새 버전을 게시하고, 시행계획 문서와 이전 규칙은 `SUPERSEDED`로 보존되는지 확인했다.
- 실제 대학 자료·학생 자료·운영 게시 상태는 사용하거나 변경하지 않았다.

### B. PostgreSQL 지표 수집 준비

- 목적/범위: 운영 변경 없이 `pg_stat_statements`와 최소 지표 수집 절차를 검토한다.
- 사전 조건: PostgreSQL 17 운영 여부 확인, 확장 허용 정책, 모니터링 계정 설계.
- 최소 권한: 합성 로컬 DB에서는 superuser, 운영에서는 별도 모니터링 역할의 읽기 권한만.
- 사전 점검: `SHOW server_version`, `SHOW shared_preload_libraries`, `pg_extension` 조회.
- 승인 게이트: preload 변경·재시작·확장 생성 승인.
- 설정 초안(실행 금지): `shared_preload_libraries='pg_stat_statements'`, 재시작 후 `CREATE EXTENSION pg_stat_statements`.
- 조회 초안(실행 금지): `calls`, `total_exec_time`, `mean_exec_time`, `rows`, `shared_blks_read`, `temp_blks_written`와 상위 지연 fingerprint 조회.
- 수집: 1분 주기, 원시 쿼리 텍스트·bind 값 미보관, 집계 fingerprint만 보관. 보관기간은 정책 승인 전 숫자를 확정하지 않는다.
- 검증: p50/p95/p99, 호출 수, 평균·총 실행시간, 반환 행, 블록·임시 블록, 커넥션 풀 대기.
- 실패 기준: 개인정보·bind 값 노출, 재시작 실패, 허용 부하 초과.
- 롤백: 확장 설정 원복·수집기 중지. 운영에서는 별도 변경창 필수.
- 완료 증거: 설정 diff·권한 목록·합성 지표 샘플·비노출 검사.

### C. 키 회전 후 검증 준비

- 목적/범위: 실제 키 없이 Fernet 암호문 구키→신키 전환과 실패 복구를 합성값으로 검증한다.
- 사전 조건: placeholder 두 개, 격리 DB, 암호문 백업.
- 최소 권한: 합성 DB와 테스트 secret만; secret manager 운영 권한 없음.
- 사전 점검: `BYOK_MASTER_KEY` 누락·공백·잘못된 형식 오류가 평문 없이 반환되는지 확인.
- 승인 게이트: 신키 병행 적용, 애플리케이션 재시작, 구키 폐기 각각 별도 승인.
- 절차 초안: 구키 복호화→신키 암호화→원자적 교체→로그 비노출→세션/CSRF/DB/외부 어댑터 smoke.
- 검증: 구키 폐기 후 구키 복호화 거부, 신키 정상 동작, 암호화 데이터 호환, 응답·로그 secret 부재.
- 실패 기준: 평문 출력, 부분 교체, 기존 암호문 손실, 인증·세션 오류.
- 롤백: 구키 암호문 백업을 격리 DB에서 복원하고 신키 폐기를 보류한다.
- 완료 증거: 합성 테스트 결과와 암호화 버전·마스킹 힌트만 기록.

### D. 보유기간 삭제 dry-run 준비

- 목적/범위: 정책으로 주입된 기준일 이전 자료의 예상 건수만 조회한다.
- 사전 조건: 공식 보유기간 정책·예외 목록·삭제 대상 테이블 승인. 현재 저장소에는 정책 숫자가 없다.
- 최소 권한: 읽기 전용 DB 계정.
- 사전 점검: 테이블·FK·cascade·created_at/confirmed_at 존재 여부와 임시 업로드 경로를 확인한다.
- 승인 게이트: 정책 담당자와 법무/개인정보 담당자의 기준일·예외 승인.
- 조회 초안(실행 금지, 파라미터 주입): `SELECT count(*)`, `min(created_at)`, `max(created_at)`와 연관 테이블별 count, `EXPLAIN (FORMAT JSON)`.
- 예외: 규칙·계산 trace·게시 결과·감사 로그는 정책상 보존 여부를 먼저 확정한다.
- 검증: 예상 건수·날짜 분포·FK 영향·인덱스 사용. 원문 식별자 출력 금지.
- 실패 기준: 기준일 미입력, 정책 예외 불명확, full scan·잠금 위험.
- 롤백: dry-run은 변경이 없으므로 롤백 불필요; 실제 삭제는 별도 런북과 백업 필요.
- 금지: DELETE/TRUNCATE/DROP/실제 DB 연결.
- 완료 증거: 비식별 집계와 EXPLAIN 결과만 보관.

### E. 공통 완료 게이트

- 실행 명령: `UV_CACHE_DIR=/tmp/uv-cache make check`, `python3 scripts/check_sensitive_data.py`.
- Playwright는 사용자 행동 후 화면·저장 데이터·콘솔 오류·A4 출력까지 검증하며, 환경변수 미설정 시 skip 사실을 기록한다.
- 담당자 서명: 실행자, 검수자, 운영 승인자, 정책 승인자, 실행/승인 시각, 변경 티켓.

## 3. 현재 판정

문서·합성 리허설 준비는 조건부 통과다. 운영 키·보유기간 정책·운영 PostgreSQL 권한·최종 공식 모집요강이 없으므로 운영 실행과 게시 전환은 차단한다.
