# 요구사항 기준

요구사항 원문은 저장소 루트의 다음 문서를 그대로 사용한다.

1. `2027_전문대_입시상담앱_실행개발문서_v2.md`
2. `CODEX_MASTER_PROMPT_2027_전문대_입시상담앱_v3.md`

v3는 v2를 상위 실행 명세로 명시한다. Phase 0에서는 저장소·자료 안전화를 우선하며, 최소 실행 화면은 도메인 기능이 없는 상태 확인용 앱 셸로 제한한다.


## 사용자 확정 지침

- 2026-07-10: 개발·운영 기본 DB는 PostgreSQL을 사용한다.
- `web` 컨테이너, `db` 컨테이너, DB 저장 volume을 분리한다.

## 프론트엔드 확정 지침

- Flask Jinja2 SSR을 기본으로 하며 SPA 프레임워크와 TypeScript를 사용하지 않는다.
- 핵심 기능은 JavaScript 없이 폼 제출과 서버 렌더링으로 동작한다.
- Vanilla JavaScript와 fetch API는 필요한 부분 상호작용에만 사용한다.
- 학생용·교사용 화면은 별도 A4 인쇄 CSS를 사용한다.

## Phase 13 복수 학과 상담

- 상담 입력은 학생 사실·성적을 한 번 준비하고 모집학년도와 복수 `program_ids`를 받는다.
- 서버는 선택한 대학·캠퍼스·학과의 해당 학년도 전형을 모두 확장해 자격을 독립 판정한다.
- `ELIGIBLE`·`CONDITIONALLY_ELIGIBLE`만 성적 자료 조회와 반영 평균등급 계산으로 진행한다.
- 핵심 결과는 배점 환산점수가 아닌 규칙별 `대학별 반영 평균등급`이며, 학과별 공개 평균등급은 같은 척도·업무키·규칙 버전일 때만 직접 비교한다.
- 결과와 학생용·교사용 A4는 대학→학과→전형 순서를 유지하고 준비 중·검토 필요·항목 오류를 숨기지 않는다.
- 모집학년도는 현재 `AdmissionRound`를 통해 전형과 연결되므로, 해당 학년도 전형이 하나 이상 있는 학과만 선택 목록에 노출한다. 전형은 있으나 게시 규칙이 없는 학과는 숨기지 않고 `계산 기준 준비 중`으로 표시하며, 전형이 전혀 없는 학과를 임의 학년도에 귀속하지 않는다.
- 공개 입시결과가 동일 등급 척도이면 과거 규칙·연도 차이를 `참고용`으로 표시할 수 있지만, 배점 등 다른 `score_basis`의 숫자는 공개 평균등급으로 표시하거나 AI payload에 전달하지 않는다.
